## Goal

Add a real **accessibility-tree perception layer** to the Computer skill — `get_window_state` (AX) and `get_accessibility_tree` — so the agent can read a window's semantic UI structure and address controls by **element index** instead of raw pixels. This recovers the Session 10 §4 diagram's "Perception: tree → shortlist" band and the semantic `scan → act → verify` loop, which the shipped vision-only implementation lacks (the core divergence recorded in #3, and the single biggest ❌ in #4's coverage).

> This is the **substantial** gap, not a small handler like #5. It changes the perception model from screenshot-only to tree-first (with vision as fallback) and unlocks element-addressed `click`/`type` and `set_value`.

## Background — what's available (verified 2026-06-14, Windows 11)

There is **no zero-cost path** (unlike #5's `shell.run`), and **no command-line shortcut** either:

- **cua CLI checked.** `cua do snapshot` returns an *AI screen summary + interactive elements* — vision/model-based (needs `ANTHROPIC_API_KEY`), i.e. the **same approach as our existing vision layer**, not a UIA tree. `cua do window ls` only lists windows (already shipped in #5). There is **no `get_accessibility_tree` / UIA command**.
- cua SDK: `Window` interface = `get_active_title` only; `Screen` = `screenshot`/`size`. No tree.
- `cua_auto`: window-manager metadata only (no AX tree).
- Installed today: `win32com`, `win32gui` (pywin32), `psutil`. **Not** installed: `uiautomation`, `pywinauto`, `comtypes`.

A modern Windows UI Automation (UIA) tree walk needs the `UIAutomationCore` COM interface, reachable via **`comtypes`** and wrapped by either:

| Library | Notes |
|---|---|
| **`uiautomation`** (recommended) | Lightweight, pip-only, wraps `comtypes`; gives a control tree, names/roles/values, and control patterns (Invoke, Value, Toggle…). |
| `pywinauto` | Heavier, broader app-driving toolkit; also pulls `comtypes`. |
| PowerShell `System.Windows.Automation` via `shell.run` | Possible (mirrors #5's pattern) but a tree-walk script is heavy and fragile — not recommended. |

So this gap **requires adding a dependency** — a project decision (see Open decisions).

## Scope

1. New module `code/computer/perception.py`: walk a window's UIA tree for a `(pid|title)` and produce (a) a Markdown/element list where every actionable control is tagged `[index N] <ControlType> "Name"`, and (b) an in-process **element-index cache** mapping `N → UIA element` for the current scan.
2. New tools in `code/computer/tools.py`:
   - `get_window_state(pid|title, mode="ax")` → element list + cache.
   - `get_accessibility_tree()` → desktop-level snapshot (apps + top-level windows).
   - element-addressed actions: `click(element_index)`, `type(element_index, value)`, `set_value(element_index, value)` — alongside the existing coordinate forms.
3. Wire it into `ComputerSkill` as the **a11y layer** of the cascade: deterministic → **a11y (tree → shortlist → element action)** → vision fallback. Honour the cache-invalidation invariant (re-scan after every state-changing action).
4. Update #4's coverage (`get_accessibility_tree`/`get_window_state`/`set_value` → ✅) and the #3 "Perception: tree → shortlist" row.

## Acceptance criteria

- [ ] `get_window_state` returns a non-empty element list for a native app (e.g. Calculator) with `[index]`-tagged controls and a populated element-index cache.
- [ ] `click(element_index)` and `set_value`/`type(element_index, …)` drive Calculator `7 × 8 = 56` **via element indices** (no pixel coordinates), verified by re-scanning the display.
- [ ] `get_accessibility_tree` returns the desktop snapshot (running apps + top-level windows).
- [ ] Cache invariant respected: an `element_index` is valid only for the latest scan; the skill re-scans after each state-changing action.
- [ ] Cascade updated so the a11y layer runs before vision; vision remains the fallback when the tree is empty (Electron/Chromium/canvas/games).
- [ ] Coverage updated in #4 (3 rows → ✅) and the #3 perception row reconciled.
- [ ] Unit smoke for the tree parser (element-list formatting + cache lookup) against a fake UIA node, plus a live Calculator element-addressed run.

## Open decisions

- [ ] **Dependency choice:** `uiautomation` (recommended) vs `pywinauto` vs PowerShell-UIA. Adds a new dep — confirm before implementing.
- [ ] Where the cache lives (per-skill-run vs per-connection) and how long an `element_index` stays valid.

## Caveats

- **Windows-only (UIA).** macOS (AX) / Linux (AT-SPI) would need their own backends — out of scope; gate on `get_environment()`.
- **Electron/Chromium/canvas/games stay opaque** to UIA — those still fall through to the vision layer (and would need CDP for DOM, tracked separately, like the Rust driver's `page` tool).
- This does **not** make us the Rust `cua-driver`; it builds the equivalent perception layer on top of the SDK. Full 34-tool/socket parity remains the migration path in #3.
